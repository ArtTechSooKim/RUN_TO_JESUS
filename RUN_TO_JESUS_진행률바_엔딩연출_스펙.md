# 전체 진행률 바 & 엔딩 골인 연출 — 기능 스펙

> 이 문서는 (1) Figma에 이 UI를 반영할 때, (2) 이후 어떤 AI(Claude Code 등)나 사람이 이 기능을 이어받아 개발할 때 참고하는 단일 스펙 문서입니다.
> `RUN_TO_JESUS_앱_개발문서.md`(프로덕션 앱 전체 기준 문서)에 추가되는 신규 기능이며, 기존 문서의 내용과 충돌하지 않습니다.

---

## 1. 의도 (한 줄 요약)

전체 진행률을 "바통을 든 러너가 결승선을 향해 달리는" 바 형태로 상시 노출하고, 행사 마지막에 참가자 전원이 본당에 모였을 때 스탭이 버튼 하나로 그 러너를 결승선까지 골인시키는 연출을 재생한다.

두 단계로 나뉜다:
1. **평상시(진행 중)**: 실제 태그 데이터에 따라 바가 조금씩 실시간으로 오름
2. **엔딩(트리거)**: 그 순간 값이 얼마든 상관없이 강제로 100%까지 차오르며 골인 — 기존에 확정된 "하이퍼스페이스 역재생 → 본당이 결승선이었다는 반전" 연출과 같은 타이밍에 맞물림

---

## 2. UX 플로우

### 2-1. 평상시 (스테이션 탐험 중)
- 태그 이벤트가 하나 발생할 때마다(`tag_events`에 행 추가) 전체 진행률이 아주 조금씩 오름
- 러너 아이콘이 바 위에서 왼쪽→오른쪽으로 위치 이동, 오른쪽 끝엔 고정된 깃발(결승선)
- 참가자 입장에서는 "내가/우리 팀이 방금 한 태그가 전체 진행률에 반영됐다"는 걸 느끼는 게 목적 — 개인 성취가 아니라 공동체 전체가 함께 달리고 있다는 감각

### 2-2. 엔딩 트리거 (행사 마지막)
1. 스탭이 기존 `/superadmin`의 "게임 종료" 버튼을 눌러 참가자 화면을 잠금 (기존 기능, 변경 없음)
2. 참가자 전원이 11층 본당에 집결
3. 스탭이 `/superadmin` **같은 화면에 새로 추가되는 "엔딩 시작" 버튼**을 누름
4. 그 순간 실제 값이 몇 %였든 상관없이, 바가 100%까지 애니메이션으로 차오르며 러너가 결승선을 통과
5. 이 타이밍이 기존 엔딩 영상 시퀀스(하이퍼스페이스 역재생 → 러닝크루 결승선 질주 → 본당 반전)와 겹치도록 스탭이 수동으로 타이밍을 맞춤

> "게임 종료"와 "엔딩 시작"은 **서로 다른 두 개의 버튼**이다. 종료는 참가자 화면을 잠그는 것, 엔딩 시작은 골인 연출을 재생하는 것 — 순서상 종료가 먼저, 엔딩 시작은 그 이후 스탭 재량으로 누르는 별도 액션.

## 3. 화면별 반영 위치

| 화면 | 반영 내용 |
|---|---|
| `/map` (참가자 메인) 상단 | 평상시 진행률 바 상시 노출 (실시간 값) |
| `public/broadcast.html` (본당 상영용) | 평상시엔 동일한 바 노출, **엔딩 시작 트리거의 주 대상** — 큰 화면에서 골인 연출 재생 |
| `/superadmin` | 기존 화면에 "엔딩 시작" 버튼 1개 추가 (게임 종료 버튼과 나란히, 별개 동작) |

> 참가자 개인 기기는 이미 "게임 종료" 시점에 화면이 잠기고 "본당으로 돌아와주세요" 메시지가 뜨므로, 엔딩 골인 연출은 기본적으로 **본당 대형 화면(`broadcast.html`)에서만 재생**하는 것을 기본값으로 함. (참가자 기기에도 동시 재생할지는 8절 열린 질문 참고)

## 4. 비주얼 스펙 (Figma 반영용)

### 정상 상태 (progress bar, idle)
- 가로로 긴 알약(pill) 형태 트랙, 둥근 모서리(반지름 = 바 높이의 절반)
- 트랙 배경: 중립 회색조 (예: surface 톤), 채워지는 부분: 액센트 컬러 1색
- 왼쪽 끝: 채워진 영역의 끝 지점에 원형 러너 배지 — 안에 "달리는 사람" 아이콘, 배지는 흰색/서페이스 배경 + 얇은 테두리
- 오른쪽 끝: 깃발 아이콘 고정 (결승선), 항상 같은 위치
- 하단 또는 좌우에 텍스트: 왼쪽 "START" 또는 현재 %, 오른쪽 "GOAL"
- 값이 바뀔 때: 채움 너비 + 러너 위치가 함께 부드럽게 트랜지션 (약 0.6~0.8초, ease)

### 엔딩 상태 (골인 애니메이션)
- 트리거 시점 값 → 100%까지 채움/러너 위치가 애니메이션으로 이동 (동일 트랜지션, 필요시 조금 더 느리게 1초 내외)
- 러너가 깃발 위치에 도달하는 순간 "GOAL" 텍스트/메시지 노출 (트랜지션 종료 후 약간의 딜레이를 두고 페이드인)
- 색상/형태는 정상 상태와 동일한 컴포넌트를 그대로 사용 — 별도의 새 UI 컴포넌트를 만들 필요 없음, **상태(state)만 다른 값**

### Figma 파일 반영 시 참고
- 기존 Figma Make 파일(`ACEHFzIJJGx6ZzAGI2Vn50`)의 스테이션 카드/대시보드 헤더 스타일 톤(플랫, 그라데이션 없음, 얇은 보더)을 그대로 따라가면 됨
- 이 바는 새로운 컴포넌트이므로 "Progress Bar / Runner"라는 이름으로 별도 컴포넌트 등록 추천 — variant 2개: `state=progress`(퍼센트 prop), `state=ending`(고정 100%, GOAL 텍스트 노출)

## 5. 프론트엔드 구현 스펙

### 공통 컴포넌트
- `RunnerProgressBar` (가칭) — props: `percent: number`, `mode: 'progress' | 'ending'`
- 내부 상태: 러너/필 위치는 `percent`에 바인딩, CSS 트랜지션으로 부드럽게 이동
- `mode='ending'`일 때는 `percent` prop을 무시하고 강제로 100%로 렌더 + GOAL 텍스트 표시 트리거

### `/map` (참가자 앱)
- 상단에 `RunnerProgressBar` 배치, `percent`는 기존 8초 폴링 중인 `/api/stats/overall` 응답값을 그대로 사용
- `mode`는 항상 `'progress'` (참가자 화면은 게임 종료 시점에 이미 잠기므로 ending 모드 진입 불필요 — 6절 데이터 흐름 참고)

### `broadcast.html` (독립 정적 페이지, RN 앱 아님)
- 기존에 있는 전체 진행률 게이지를 이 `RunnerProgressBar` 형태로 교체 (바닐라 JS/CSS로 동일 컴포넌트 재구현 — RN 코드와 공유는 안 되므로 시각 사양만 동일하게 맞춤)
- 폴링 중 `game_state`가 `'ending'`으로 바뀐 걸 감지하면 `mode='ending'`으로 전환, 100%로 애니메이션 후 GOAL 표시
- 한 번 ending 모드에 진입하면 되돌리지 않음 (페이지 새로고침 전까지 유지)

### `/superadmin`
- 기존 "게임 종료" 버튼 옆에 "엔딩 시작" 버튼 추가
- 클릭 시 확인 모달(기존 종료 버튼과 동일한 패턴) → 확인 시 API 호출

## 6. 백엔드 구현 스펙

### `app_state` 테이블 확장
기존: `app_state(id=1, game_state)` — 값은 `progress` / `ended` 두 가지였음
**변경**: `game_state`에 세 번째 값 `ending` 추가

```
game_state: 'progress' | 'ended' | 'ending'
```

- `progress`: 평상시, 참가자 화면 정상 동작
- `ended`: 기존 그대로 — 참가자 화면 잠금 + "본당으로 돌아와주세요" 메시지
- `ending`: 신규 — 참가자 화면 잠금 상태는 유지(= `ended`와 동일한 잠금 UX), **다만 `broadcast.html`은 이 상태를 감지해서 골인 애니메이션 재생**

> `ending`은 `ended` 이후에만 진입 가능하도록 서버에서 검증 (순서 강제: progress → ended → ending). 되돌릴 수 없는 단방향 전이.

### 신규 API
```
POST /api/admin/ending/start
```
- 최고관리자 인증 필요 (기존 슈퍼어드민 체크와 동일한 방식)
- 현재 `game_state`가 `ended`일 때만 허용, `ending`으로 갱신
- 응답: 갱신된 `game_state`

### 클라이언트 폴링
- `broadcast.html`은 기존에 `/api/stats/overall`을 폴링 중이라면, 여기에 `game_state`도 같이 내려주는 형태로 응답 확장 (또는 기존 `app_state` 조회 API를 함께 폴링)
- `game_state === 'ending'`을 감지한 첫 시점에 로컬에서 `mode='ending'`으로 전환 — 서버는 퍼센트를 100으로 바꿀 필요 없음, **트리거 신호만 주면 되고 100%까지 차오르는 건 클라이언트 애니메이션**

## 7. 상태 다이어그램 (텍스트)

```
[progress] --(스탭: 게임 종료 버튼)--> [ended] --(스탭: 엔딩 시작 버튼)--> [ending]
```

- `progress → ended`: 기존 기능, 변경 없음
- `ended → ending`: 신규, 단방향, 되돌릴 수 없음
- `ending` 상태에서 "모든 팀 진행도 리셋"을 실행하면 `progress`로 완전히 되돌아감 (기존 리셋 기능 그대로 재사용, 별도 처리 불필요)

## 8. 기존 엔딩 시퀀스와의 통합 지점

- 확정된 엔딩 연출(하이퍼스페이스 역재생 → 러닝크루 결승선 질주 → 본당 반전 → RUN TO JESUS 10글자 점등 → 히브리서 선포)은 **영상 재생**으로 진행되는 부분이고, 이 진행률 바 골인은 **`broadcast.html` 상의 실시간 UI 애니메이션**이라 서로 다른 채널
- 두 연출의 타이밍은 자동 동기화되지 않음 — 스탭이 영상 재생 버튼과 "엔딩 시작" 버튼을 **수동으로 원하는 타이밍에 각각** 누르는 방식 (7/19 리허설에서 두 버튼을 누르는 순서/간격을 실제로 맞춰볼 것)
