# RUN TO JESUS — 스테이션/조각 재배치 구현 가이드

Claude Code(VSCode)가 `RUN_TO_JESUS` 저장소에서 이 변경사항을 반영할 때 참고할 스펙 문서입니다.
로컬 개발은 프로덕션 DB에 직접 씀 → 마이그레이션/시드 스크립트 실행 전 반드시 사람에게 확인받을 것.

---

## 1. 최종 확정 배치 (Source of Truth)

| 방 | 장소 | 조각 | 모자이크 칸(position) |
|---|---|---|---|
| 라합방 | 사무엘홀+초등부교사실(세션1) / 다니엘홀+유년부교사실(세션2) | R | 1 |
| 노아방 | 플레이그라운드 | U | 2 |
| 요나방 | 다윗홀 | N | 3 |
| 요셉방 | 요셉홀 | T | 4 |
| 야곱방 | 이삭홀 | O | 5 |
| 아브라함방 | 아가페홀 | J | 6 |
| 삼손방 | 디모데홀 | E | 7 |
| 도미노 | 새로운홀 | S | 8 |
| 영화관 | 여호수아홀 | U | 9 |
| 숨은글자찾기 | 미정(당일 QR만 부착) | S | 10 |

RUN TO JESUS 10글자 = R-U-N-T-O-J-E-S-U-S. U와 S는 각 2번씩 등장하므로 **글자만으로는 모자이크 칸을 특정할 수 없음** → `fragment_letter`와 별개로 `fragment_position`(1~10)을 반드시 저장해야 함.

## 2. 기존 상태에서 바뀐 것 (diff)

- **블러핑** → **야곱방** (이삭홀 유지, 조각 O 유지)
- **아벨방**(장소 미정) → **요나방** (장소 다윗홀로 확정, 조각은 이번에 N 배정)
- **미정게임** → **영화관** (여호수아홀 유지)
- **삼손방** 장소: 디모데홀(원안) — 중간에 프레이즈홀로 잘못 정정됐다가 디모데홀로 최종 환원. 코드/문서에 프레이즈홀이 남아있는 곳이 없는지 확인.
- **라합방**: 기존 R,U,N 3글자 보유 → **R 1개만** 보유. U,N은 노아방·요나방으로 이동.
- **도미노**: 기존 S,U,S 3글자(모자이크 2칸) 보유 → **S 1개만** 보유. U는 영화관으로, S 1개는 신규 방 "숨은글자찾기"로 이동.
- **신규 스테이션 추가**: 숨은글자찾기 (담당 FD: 김수, 장소 랜덤/당일 결정, QR 전용, NFC 없음)

---

## 3. 숨은글자찾기 스테이션 — 특수 처리 규칙

일반 스테이션과 다르게 취급해야 함:

1. **station 마스터 데이터에는 등록하되, 지도/스테이션 목록 UI에는 노출하지 않음.**
   - `stations` 테이블에 row는 생성 (station_id, name="숨은글자찾기", venue=NULL 또는 "TBD", is_hidden=true 같은 플래그 권장)
   - 앱 내 지도 컴포넌트, 스테이션 리스트 화면에서 `is_hidden` true인 항목은 필터링해서 안 보이게 처리
2. **NFC 아님, QR 전용.** 기존 QR 스캔 플로우 그대로 재사용 (새 스캔 로직 불필요).
3. **장소가 당일 유동적**이므로 `venue` 필드를 하드코딩하지 말 것. QR 코드 자체에 station_id만 인코딩되어 있으면 충분 — 물리적으로 어디 붙어있는지는 앱 로직과 무관.
4. **동선/지도 표시 로직에서 완전히 제외.** 아래 4번 항목(지도 갱신) 작업 범위에도 포함하지 않음.

## 4. 데이터베이스 변경사항

### 4.1 `stations` 테이블 (또는 현재 station 마스터 데이터 구조)
- 야곱방, 요나방, 영화관으로 이름 변경 (기존 블러핑/아벨방/미정게임 row 업데이트, 또는 신규 row + 기존 row deprecate)
- 삼손방 venue → 디모데홀로 확정
- 숨은글자찾기 신규 row 추가 (`is_hidden = true`)

### 4.2 `STATION_ALIASES` (레거시 NFC 태그 리다이렉트용)
기존 NFC 태그가 이미 물리적으로 부착되어 있다면, 옛 이름으로 스캔됐을 때 새 station_id로 매핑되도록 별칭 추가:
```
블러핑 → 야곱방
아벨방 → 요나방
미정게임 → 영화관
```
이미 존재하는 STATION_ALIASES 패턴(RAHAB 사례)을 그대로 따를 것.

### 4.3 fragment 매핑 테이블 (신규 또는 기존 컬럼 확장)
`station_id → fragment_letter` 단일 매핑이었다면, `fragment_position` 컬럼을 추가해 아래처럼 확장:

| station_id | fragment_letter | fragment_position |
|---|---|---|
| 라합방 | R | 1 |
| 노아방 | U | 2 |
| 요나방 | N | 3 |
| 요셉방 | T | 4 |
| 야곱방 | O | 5 |
| 아브라함방 | J | 6 |
| 삼손방 | E | 7 |
| 도미노 | S | 8 |
| 영화관 | U | 9 |
| 숨은글자찾기 | S | 10 |

`tag_events` insert 시 이 매핑을 참조해 `fragment_position`도 같이 기록.

---

## 5. 모자이크(엔딩 화면) 로직 수정

- 기존에 `fragment_letter` 기준으로 글자별 밝기를 계산했다면, 이제 **`fragment_position`(1~10) 기준**으로 바꿔야 함. 그렇지 않으면 U/S 두 칸이 서로의 진행률을 덮어씀.
- `global_ratio` 계산(해당 station을 방문한 팀 수 / 24)은 station_id 기준 그대로 유지, 다만 그 비율을 반영할 대상 칸을 `fragment_position`으로 지정.
- 숨은글자찾기(10번 칸)도 다른 칸과 동일한 밝기 로직 적용 — 특수 취급 불필요 (지도 노출만 예외).

---

## 6. 이번 작업 범위에서 제외되는 것

- 숨은글자찾기의 지도 표시 (의도적으로 안 함)
- 숨은글자찾기의 동선 설계 (담당자 재량)
- 아브라함방 관련 변경 없음 (이름/장소/조각 전부 기존 유지)
