# RUN TO JESUS: The Race Through Time — 앱 개발 문서

> 이 문서 하나만 읽으면 아무나 이 프로젝트를 처음부터 이어받아 개발할 수 있도록 작성. 2026-07-25 실제 행사에서 쓸 프로덕션 앱 기준 (초기 데모/프로토타입 단계 문서 아님 — 이전 버전은 시연용 계획서였고, 지금은 실사용 앱이 이미 만들어져 배포돼 있음).

---

## 1. 프로젝트 개요

- **행사명**: RUN TO JESUS: The Race Through Time
- **성격**: 새로운교회 청년 연합수련회 (주니어청년/청년1/청년2 공동체 최초 합동 수련회), **약 300명, 24개 팀** 규모
- **행사일**: 2026-07-25
- **핵심 메시지**: 히브리서 12:1-2 — 믿음의 경주는 세대를 넘어 이어지며, 우리는 그 바통을 다음 세대에 전하는 증인이다
- **앱 목적**: 참가자들이 팀 단위로 공간(스테이션)을 탐험하며 NFC 태깅/QR 스캔으로 "믿음의 조각"(RUN TO JESUS 10글자)을 모아 최종 메시지를 완성하는 이머시브 체험 보조 도구
- **현재 상태**: 실제 행사에서 쓸 앱이 이미 만들어져 Railway에 배포되어 있고, 계속 다듬는 중. 아래 내용은 지금 실제로 동작하는 기준으로 작성됨

## 2. 기술 스택 (현재 기준)

- **프레임워크**: Expo SDK 57 + Expo Router (`~57.0.1`, `expo-router ~57.0.2`, `react-native 0.86.0`)
  - ⚠️ Expo가 최근 크게 바뀌었으므로, 코드 작성 전 [Expo v57 versioned docs](https://docs.expo.dev/versions/v57.0.0/) 확인 권장 (`AGENTS.md` 참고)
- **애니메이션**: `react-native-reanimated`
- **오디오**: `expo-audio` (구 `expo-av`는 더 이상 안 씀)
- **NFC**: `react-native-nfc-manager` (v4 beta), **QR 스캔**(`expo-camera`)도 병행 지원 — 둘 중 하나로 태깅 가능, 안 되면 관리자/참가자가 수동 완료 처리 가능
- **백엔드**: Express (`server/index.js`, `server/routes.js`) + MySQL (`server/db.js`, `mysql2/promise`)
  - Express 프로세스 하나가 정적 웹 빌드(`dist/`)와 `/api/*` 라우트를 **동시에** 서빙함 (SPA 폴백 포함)
- **배포/호스팅**: Railway — 서비스 2개: `run-to-jesus`(앱+API), `MySQL-3uIX`(DB)
- **디자인 출처**: Figma Make 파일 (`ACEHFzIJJGx6ZzAGI2Vn50`, "Define User Flow") — 단, 지금은 실제 앱이 이 파일보다 많이 앞서 나가 있어서 참고용일 뿐, 그대로 베끼면 안 됨 (10절 참고)

### ⚠️ 배포 시 반드시 알아야 할 것

**`git push`는 배포되지 않습니다.** 이 Railway 서비스는 GitHub와 연결되어 있지 않고, 오직 `railway` CLI로만 배포됩니다. 코드를 GitHub에 푸시한 뒤 **반드시** 아래를 추가로 실행해야 실제로 반영됩니다:

```
railway up --service run-to-jesus --detach
```

그 다음 `railway deployment list --service run-to-jesus --json`으로 최신 배포의 `status`가 `SUCCESS`가 될 때까지 확인 (빌드에 보통 2~3분 소요 — 매번 `npx expo export -p web` 전체를 다시 빌드함).

### 로컬 개발 시 주의

이 프로젝트에는 **로컬 백엔드가 없습니다.** `src/lib/api.ts`의 `API_BASE_URL`은 항상 프로덕션 Railway URL을 가리킵니다. 즉 로컬에서 `npx expo start --web`으로 개발 서버를 띄워도 **실제 프로덕션 DB에 직접 쓰기/읽기**를 합니다. 테스트할 때 생긴 데이터(태그 이벤트, 세션 등)는 테스트 후 꼭 정리할 것.

## 4. 데이터 모델 (MySQL, `server/db.js`)

| 테이블 | 용도 |
|---|---|
| `users(person_id UUID PK, name, team_id)` | 첫 로그인 시 자동 생성 (가입 절차·명단 검증 없음). 이름 중복 시 자동으로 숫자 붙임 (김수 → 김수2 → 김수3) |
| `stations(station_id PK, name, hall_name, duration_minutes, concurrent_capacity, letters JSON, is_active)` | `src/constants/stations.ts`의 `STATION_SEED`를 그대로 미러링 — **두 곳을 항상 같이 수정할 것** |
| `tag_events(person_id, team_id, station_id, fragment_letter, tagged_at)` | **팀 진행도의 유일한 진실**. 글자가 여러 개인 스테이션은 글자 수만큼 행이 들어감 |
| `fragment_reveal_log` | 통계 전용, best-effort. 정확성 판단에는 안 씀 |
| `game_sessions(id, station_id, team_id, started_at, expected_end_at, status, ended_at, ended_by, started_by_name)` | "지금 이 스테이션에 어느 팀이 있는지". `GET /sessions` 호출 때마다 만료된 세션을 지연 처리(백그라운드 타이머 없음) |
| `app_state(id=1, game_state)` | 전체 게임 진행중/종료 단일 토글 |

### 핵심 설계 원칙 — 절대 사람 단위가 아니라 팀 단위

- `GET /teams/:team_id/fragments`가 팀이 모은 글자를 판단하는 유일한 기준. 팀원 몇 명이 로그인했든 상관없이, 팀 하나가 스테이션을 한 번만 태그하면 팀 전체가 클리어됨
- `/api/stats/overall`(본당 상영용 전체 달성률)의 분모도 **24(고정) × 활성 스테이션 수**로 계산 — 실제 가입 인원과 무관
- 팀원 한 명이 잘못 태그한 걸 취소하면(`DELETE /api/tag-events/:team_id/:station_id`), 팀의 다른 기기들도 8초 폴링으로 자동 반영됨

## 7. 핵심 설계 결정 (다시 논의하지 않아도 되는 것들)

- **명단 검증 없음**: 등록이 늦게 마감되고 당일 인원 변동이 있을 수 있어서, 로그인은 팀 번호/이름을 자율 입력받고 서버 검증을 하지 않음. 대신 위 4절처럼 진행도가 팀 단위라서 인원수 영향을 안 받음
- **관리자 비밀번호(`saeroun0906`)**: 행사 규모를 고려해 의도적으로 단순하게(클라이언트 하드코딩) 처리. 단, "모든 팀 진행도 리셋"만은 서버에서도 검증(파급 범위가 전체 행사 데이터라 예외)
- **최고관리자 진입**: 로그인 폼에 이름="김수"+팀="100" 입력 시 특수 케이스로 즉시 진입 (실제 로그인 API도 안 탐)
- **"팀 진실 vs 기기별 리빌"**: 각 기기는 AsyncStorage에 "이 기기가 이미 보여준 조각 리빌 애니메이션" 목록을 따로 저장해서, 팀원이 다른 기기에서 태그한 조각을 아직 못 본 경우에만 리빌 애니메이션을 재생함. 태그 취소/전체 리셋 시 이 캐시도 자동으로 정리됨

## 10. 이 문서 사용법

- 새 세션(Claude Code든 다른 AI든, 사람이든)에서 이 파일만 읽으면 프로젝트 맥락 파악 가능
- 더 기술적/AI 친화적인 요약이 필요하면 레포 루트의 `PROJECT_SUMMARY.md` 참고 (코드 경로·줄 번호까지 포함된 버전)
- 배포 전에는 반드시 2절의 "배포 시 반드시 알아야 할 것" 확인 — `git push`만 하고 끝냈다고 착각하지 말 것
